## Agent 工作规则

进入 S2 之后，每个部署步骤都必须遵守根目录 `agent.md`：

- 忠于本地文档、现场事实和实际命令输出。
- 不确定的信息必须标为未知，并写出验证方法。
- 每一步执行后都要更新 `docs/deployment_log.md`。
- 如果阶段状态、部署选择、风险或阻塞发生变化，要同步更新 `docs/current_bringup_status.md`。
- 每一步之后都要回看并矫正本行动路线，保证它反映最新事实，而不是停留在初始计划。

## 0. 总体原则

1. 先资料、再离线模型、再只读通信、再低速运动。
2. 每个阶段只打开一个控制源：Web、SDK、ROS、原始 CAN 不能并行抢控制。
3. 初次真机只做状态读取和反馈确认，不直接做点位、直线、圆弧或 MoveIt 执行。
4. 任何异常状态先停机、记录、确认原因，再清错或恢复。
5. 原始 CAN 用于协议验证和必要排障，不作为首选控制入口。
6. 双臂现场必须先建立唯一身份。未完成网络名/IP/CAN 口映射前，只按单臂逐台验机。

## 上游仓库职责边界

这三个 GitHub 仓库不是同一层东西，后续使用必须按职责分开：

| 仓库 | 本地证据路径 | 定位 | 后续使用方式 |
| --- | --- | --- | --- |
| `agilexrobotics/agx_arm_ros` | `upstream/agx_arm_ros` | 当前 NERO ROS2 集成与控制工作区。它封装 ROS topic/service/launch、RViz/MoveIt 入口，并通过 SDK/CAN 与硬件交互。 | 作为 S2B/S3/S8/S12 的主 ROS 包。正式部署时在选定 ROS2 主机的 `~/agx_arm_ws/src` 中 clone `ros2` 分支并拉取 submodule；初始使用 `arm_type:=nero`、`effector_type:=none`。 |
| `agilexrobotics/pyAgxArm` | `upstream/pyAgxArm` | Python SDK 和 CAN 硬件访问层。它不依赖 ROS，可独立读状态，也被 `agx_arm_ros` 使用。 | 作为 S2A/S7 的 SDK-only 路径。当前 Ubuntu 20.04 主机可先用它做 CAN/状态只读准备；必须满足 `python-can>=3.3.4`，固件 selector 等 Web 读到真实版本后再定。 |
| `agilexrobotics/piper_ros` | `upstream/piper_ros` | 用户给出的 NERO URDF 来源路径所在仓库，`humble_beta1` 分支上的 `nero_description` 是模型/描述参考。 | 作为模型交叉核对来源，用于比较 URDF、mesh、关节命名和历史差异；不作为默认 NERO ROS 控制工作区。 |

补充判断：

- `agx_arm_ros` 内的 `agx_arm_urdf` submodule 是当前 ROS2 驱动实际使用的模型集，优先用于正式 ROS2 工作区。
- `piper_ros` 的价值是确认用户最初提供的 URDF 来源和模型差异，不承担当前控制驱动职责。
- `pyAgxArm` 可以先在无 ROS 的当前主机上准备，但 SDK-only 通过不等于 ROS 环境通过。
- `upstream/` 是本项目证据缓存；正式运行和构建仍放在选定部署工作区，例如 `~/agx_arm_ws`。

## S1 安全与场地

**阶段目标**

达到可上电、可观察、可停机、不会因失能掉落造成伤害或设备损坏的现场条件。

**执行动作**

- 场地留出完整工作空间，NERO 工作半径约 `580 mm`，周边不得放易碰撞物。
- 限制人员进入机械臂可达区域；操作人员不得穿宽松衣物，长发需固定。
- 明确紧急停止路径：
  - Web 急停按钮。
  - CAN `0x150 byte0=0x01` 快速急停。
  - 必要时拔航插断电，但这是最后应急手段。
- 准备日志记录：拍照记录接线、状态灯、Web 版本页、CAN 抓包时间。

**验收标准**

- 机械臂固定牢靠；底座安装图已确认 `70 mm x 70 mm` 孔位和 4 个 `Ø5.4` 贯穿孔，按 M5 紧固件固定。
- 操作范围内无人、无线缆拉扯、无桌面松动物。
- 每个参与者知道急停按钮和断电位置。
- 已确认断电/失能时机械臂可能下落，末端不会砸到人或设备。

**注意事项**

- NERO 本体最大负载 `3 kg`，不要超载。
- 非定制防护下按 IP22 对待，不按防水防尘设备使用。
- 使用/存储环境按 `-20 到 50 摄氏度`、`25%-85% RH 非冷凝` 管控。
- 机械臂运行后可能发热，刚停止后不要直接长时间接触。

仍然不安装原生 ROS，不选择 Noetic，不执行真机运动命令。

## S6 CAN 只读调试

**阶段目标**

确认两条独立 SocketCAN 链路工作，能分别稳定读取 Arm A 和 Arm B 的 NERO 反馈帧。

**执行动作**

先发现 CAN 模块 USB bus-info：

```bash
bash scripts/find_can_ports.sh
```

分别激活两条 CAN：

```bash
bash scripts/activate_can.sh can_arm_a 1000000 "<Arm A USB bus-info>"
bash scripts/activate_can.sh can_arm_b 1000000 "<Arm B USB bus-info>"
ip -details link show can_arm_a
ip -details link show can_arm_b
```

只读抓包：

```bash
timeout 10s candump can_arm_a
timeout 10s candump can_arm_b
```

期望协议事实：

- CAN 标准：CAN2.0B 标准帧。
- 波特率：`1000K`。
- 数据格式：Motorola/MSB。

优先观察反馈帧：

| ID | 内容 | 验收用途 |
| --- | --- | --- |
| `0x2A1` | 机械臂状态 | 模式、状态、故障码 |
| `0x2A2-0x2A4` | TCP 位姿 | X/Y/Z/RX/RY/RZ |
| `0x2A5-0x2A7` | J1-J6 角度 | 关节反馈 |
| `0x2A9` | J7 角度 | 第 7 轴反馈 |
| `0x2A8` | 夹爪反馈 | 夹爪行程、扭矩、状态 |
| `0x251-0x257` | 关节高速反馈 | 20 ms，速度/电流/位置 |
| `0x261-0x267` | 关节低速反馈 | 100 ms，电压/温度/状态 |

**验收标准**

- `can_arm_a` 与 `can_arm_b` 均为 UP 状态，bitrate 为 `1000000`。
- `candump can_arm_a` 与 `candump can_arm_b` 均连续收到反馈帧。
- `can_arm_a` 与 `can_arm_b` 均处于 `ERROR-ACTIVE`。
- 两条链路各自能观察到 `0x2A1`、`0x2A5-0x2A7` 和 `0x2A9`。
- `0x2A1` 状态正常，无急停、无解、奇异、超限、通信异常、碰撞、过温等状态。
- J1-J7 反馈均连续，J7 使用 `0x2A9`。
- 拔掉 Arm A CAN 时只停止 `can_arm_a` 反馈；拔掉 Arm B CAN 时只停止 `can_arm_b` 反馈。

**注意事项**

- 此阶段不发控制帧，只抓包。
- 进入 S6 前，先完成两条臂各自的 S5 七个关节页签只读记录和唯一身份记录。
- 双臂首次 CAN 不共用总线，先一条臂一个 USB-CAN 单独抓包。
- 双臂同时 CAN 使用两个独立官方 USB-CAN 模块，并映射为 `can_arm_a`、`can_arm_b`。
- 当前物理 USB 口记录：Arm A 使用竖着插入的 USB 口，bus-info `1-5:1.0`；Arm B
  使用横着插入的 USB 口，bus-info `1-11:1.0`。
- 如果没有反馈，先确认 Web 端 CAN 通讯开关、CAN_H/L、波特率和供电。
- 若必须打开 Web 端 CAN 通讯开关才能收到反馈，先逐臂打开并抓包；只有在确认两条 CAN 总线完全独立后，
  才测试两个 `candump` 同时运行。
- 不要在同一条 CAN 总线上让两条机械臂都开启外网 CAN 推送。

- 执行时两臂目标和回原误差不超过 `0.7 deg`，非目标关节最大偏差不超过 `1.0 deg`。
- 现场可见运动方向必须与当前原语定义一致；局部关节正负号不能直接当成
  `lab_world` 下的方向语义。
- 每个原语都有开始状态、目标、反馈、结束状态、视频/照片和 rosbag 证据。
- 可作为后续 manipulation 算法的最小安全回归集。

**注意事项**

- S13 的“协同”只代表时间上同时受控，不代表可以做接触、交接、双臂抓取或人体附近动作。
- 当前 `30 deg` 只适用于已在 S12 分别验证过的单臂 J1 动作；S13 已证明
  单臂符号组合不能直接外推为双臂世界方向语义，不外推到其他关节、笛卡尔或末端接近动作。
- 任何需要两臂末端接近的任务都必须另写安全空间、碰撞检查和停止策略。

**阶段目标**

将 NERO 接入项目应用，形成可重复运行、可诊断、可回退的系统。

**执行动作**

  - 每次更换末端工具、安装姿态、TCP、固件、SDK/ROS 版本后，回到 S6-S10 做最小回归。

## 原始 CAN 开发闸口

原始 CAN 只在以下条件全部满足后使用：

- Web、SDK、ROS 只读链路已通过。
- 已能识别 `0x2A1` 状态和故障码。
- 已确认急停帧 `0x150 byte0=0x01` 与恢复流程。
- 所有手写帧都先在文档中写出 ID、长度、字节含义、单位和预期反馈。
- 所有设置类命令都等待 `0x476` 或对应反馈确认。

关键帧摘要：

| ID | 方向 | 用途 | 调试策略 |
| --- | --- | --- | --- |
| `0x150` | 控制到主控 | 急停、轨迹、示教 | 急停可用性优先验证 |
| `0x151` | 控制到主控 | 控制模式、MOVE 模式、速度百分比、安装位置、CAN 推送 | 模式切换必须看 `0x2A1` |
| `0x152-0x154` | 控制到主控 | TCP 位姿目标 | S10 后再用 |
| `0x155-0x157` | 控制到主控 | J1-J6 目标角 | 低速小角度 |
| `0x170` | 控制到主控 | J7 目标角 | 不要漏第 7 轴 |
| `0x159` | 控制到主控 | 夹爪控制 | 仅装夹爪后使用 |
| `0x15A-0x160` | 控制到主控 | MIT 控制 | 高风险，默认禁用 |
| `0x470` | 控制到主控 | 主从模式 | 专项审批 |
| `0x471` | 控制到主控 | 电机使能/失能 | 优先 Web/SDK/ROS |
| `0x475` | 控制到主控 | 零点、加速度、清错 | 专项审批 |
| `0x47A` | 控制到主控 | 碰撞防护等级 | 保守配置 |
| `0x422` | 控制到主控 | 固件升级模式 | 供应商指导下执行 |

## 常见异常处理方向

| 现象 | 优先检查 |
| --- | --- |
| 无 CAN 帧 | Web CAN 开关、CAN_H/L、bitrate、供电、USB-CAN 驱动 |
| 状态灯红色 | Web 首页关节状态、0x2A1 机械臂状态、低速反馈 |
| 关节不动 | 是否使能、是否当前控制模式、目标是否越限、是否奇异/无解 |
| 点位运动无响应 | 原点奇异、目标不可达、姿态不合法、未切 Web/CAN 控制模式 |
| 夹爪异常 | ID 冲突、未使能、未回零、过流/过温/传感器状态 |
| 恢复后仍不可控 | 清错、重新使能、确认模式，必要时重启系统 |
